# 데이터 저장구조 (DB 설계)

> **이 문서의 범위**: 대시보드(기획안.md v2 + 기획안_v3.md)가 라이브로 불러오는 데이터를 어떻게
> 영속 저장할지에 대한 설계. 판단값(config.json)·수동 콘텐츠(sectors.json)는 건드리지 않고,
> 그 위에 **새로운 저장 계층 하나만 추가**한다. v3의 OpenBB 전환과는 독립적 — 이 스키마는
> 데이터가 `fredapi`로 오든 `pykrx`로 오든 `obb.*`로 오든 동일하게 받는다(provenance만 `source`
> 컬럼에 다르게 기록됨).

## 0. 왜 필요한가

현재는 `st.cache_data(ttl=3600)`뿐이다 — 메모리 캐시라 프로세스가 재시작되면 사라지고, 캐시가
만료될 때마다 동일한 과거 구간(예: KOSPI 365일)을 통째로 다시 받는다. 이게 실제로 막고 있는 것들:

1. **과거 EPS 추정치 히스토리가 없음** — 기획안.md §11에서 이미 짚은 한계. Block G가 "과거 지수
   vs 오늘 입력한 EPS"인 이유가 EPS가 바뀌어온 기록 자체가 없어서다.
2. **REGIME/서브점수가 매번 재계산만 되고 저장되지 않음** — "언제 BASE에서 BULL로 바뀌었는지"
   같은 진짜 히스토리가 없다.
3. **RRG가 점 하나만 찍힌다** — 실제 RRG 차트의 표준 기능인 trailing tail(과거 N일 경로)은
   일별 RS-Ratio/Momentum이 누적 저장돼야 그릴 수 있다.
4. **v3가 요구하는 walk-forward·이벤트스터디·Triage 검증** 은 전부 "오랫동안 쌓인 일별 데이터"가
   전제인데, 지금 구조로는 누적이 전혀 안 된다.
5. **API 호출이 비효율적** — 매 캐시 만료마다 동일한 과거 데이터를 재요청한다(콜드 로드 시장당
   20~25초가 매시간 반복).

## 1. 설계 원칙 — 3계층 분리

기존에 이미 확립된 "config.json(판단값) vs sectors.json(콘텐츠) 분리" 원칙을 그대로 확장한다.

| 계층 | 예시 | 저장 위치 |
|---|---|---|
| 판단값 | 멀티플 앵커, 가중치, 임계값, RRG 윈도우 | `config.json` (변경 없음) |
| 수동 콘텐츠 | 섹터 테마 그룹, key/star 테제 | `sectors.json` (변경 없음) |
| **🆕 시계열·계산값** | FRED 시리즈, 지수/종목 가격, S01~S06 점수, 합성점수·regime, RRG, 워치리스트 발동 이력, manual override **변경 이력** | **신규 DB** |

manual_overrides의 "현재값"은 계속 config.json에 둔다(사람이 직접 읽고 고치기 쉬워야 하므로).
DB에는 그 값이 **바뀌어온 이력**만 append-only로 쌓는다 — 현재값과 이력을 분리해야 config.json이
계속 "지금 뭐가 입력돼 있나"를 한눈에 보여주는 역할을 유지한다.

## 2. 스키마

긴 형식(long format) 위주로 — 추적 시리즈/티커가 계속 늘어나는 프로젝트 특성상 시리즈마다
테이블을 새로 만드는 wide format보다 훨씬 유연하다.

```sql
-- 1. 모든 시계열의 원천값 — FRED 시리즈, 지수레벨, 개별종목가, 섹터ETF가 전부 여기 하나로
CREATE TABLE series_daily (
    series_id   TEXT NOT NULL,    -- 'WALCL' | 'KOSPI_INDEX' | '005930.KS' | 'XLK' | 'NVDA' ...
    date        DATE NOT NULL,
    value       REAL NOT NULL,
    source      TEXT NOT NULL,    -- 'fred' | 'pykrx' | 'yfinance' | 'openbb' (그 날짜 값의 출처)
    fetched_at  TIMESTAMP NOT NULL,
    PRIMARY KEY (series_id, date)
);
CREATE INDEX idx_series_daily_lookup ON series_daily(series_id, date);

-- 2. manual_overrides 변경 이력 — config.json은 "현재값"만, 여기는 전체 이력
CREATE TABLE manual_override_history (
    key         TEXT NOT NULL,    -- 'KOSPI_forward_eps' | 'credit_balance_krw' ...
    value       REAL,
    as_of       DATE,
    recorded_at TIMESTAMP NOT NULL,
    PRIMARY KEY (key, recorded_at)
);

-- 3. 일별 서브점수·합성점수·regime (지금은 계산만 되고 저장 안 됨)
CREATE TABLE scores_daily (
    market      TEXT NOT NULL,    -- 'KOSPI' | 'NASDAQ'
    date        DATE NOT NULL,
    s01 REAL, s02 REAL, s03 REAL, s04 REAL, s05 REAL, s06 REAL,
    composite   REAL,
    regime      TEXT,             -- 'base' | 'bull' | 'hyper'
    n_available INTEGER,
    PRIMARY KEY (market, date)
);

-- 4. 섹터 일별 메트릭 — RRG의 trailing tail을 만들려면 필수
CREATE TABLE sector_metrics_daily (
    market       TEXT NOT NULL,
    sector_code  TEXT NOT NULL,
    date         DATE NOT NULL,
    market_cap   REAL,
    ytd_pct      REAL,
    rs_ratio     REAL,
    rs_momentum  REAL,
    quadrant     TEXT,
    PRIMARY KEY (market, sector_code, date)
);
CREATE INDEX idx_sector_metrics_trail ON sector_metrics_daily(market, sector_code, date);

-- 5. 워치리스트 발동 이력 — v3가 요구하는 confluence 빈도/정밀도 실측에 직접 사용
CREATE TABLE watchlist_events (
    trigger_id   TEXT NOT NULL,
    date         DATE NOT NULL,
    status       TEXT NOT NULL,   -- 'fired' | 'quiet' | 'manual_check' | 'unknown'
    recorded_at  TIMESTAMP NOT NULL,
    PRIMARY KEY (trigger_id, date)
);
```

별도 "leaders/key-star 가격" 테이블은 만들지 않는다 — key/star 티커도 결국 `series_daily`에
`series_id='005930.KS'` 같은 행으로 들어가므로 중복 테이블이 불필요하다.

## 3. 저장엔진: SQLite

**권장: SQLite.** Python stdlib(`sqlite3`)라 새 의존성이 0개, pandas가 `to_sql`/`read_sql_query`를
네이티브로 지원, 단일 파일(`dashboard.db`)이라 config.json·sectors.json과 같은 "로컬 단일앱"
철학을 그대로 유지한다. 이 프로젝트 데이터량(시리즈 ~100개 × 일별 × 몇 년 = 수만~수십만 행)에는
전혀 부담이 없다.

**대안: DuckDB.** v3의 walk-forward 백테스트처럼 윈도우 집계·analytical 쿼리가 많아지면 SQLite보다
빠르다. 스키마가 동일한 SQL이라 전환 비용이 낮으므로, 지금 단계에서 바로 DuckDB로 가는 건 과한
선택이라 판단했다 — **백테스트가 실제로 이 DB를 직접 두드리게 될 때 재검토**.

**Postgres 등 서버형 DB는 채택하지 않음** — "로컬 Python 단일앱"이라는 기존 기술스택 결정(기획안.md
§9)과 맞지 않고 운영 부담만 늘어난다.

## 4. 수집 패턴: "전체 재요청" → "증분 fetch + upsert"

현재: `cached_kospi_level_series(365)` 같은 함수가 캐시 만료마다 365일 전체를 다시 받는다.

이후:

```python
def ensure_series_up_to_date(series_id, fetch_fn, bootstrap_lookback_days=365):
    latest = db.query_latest_date(series_id)
    if latest is None:
        df = fetch_fn(lookback_days=bootstrap_lookback_days)   # 최초 1회만 전체 부트스트랩
    else:
        gap_days = (today() - latest).days
        if gap_days <= 0:
            return                                             # 오늘 이미 최신
        df = fetch_fn(lookback_days=gap_days + 2)               # 여유 2일치만
    db.upsert_series(series_id, df, source=...)
```

1. DB에서 해당 `series_id`의 최근 날짜 조회
2. 그 다음날부터 오늘까지만 API로 fetch (대부분 1~수일치)
3. `series_daily`에 upsert
4. 차트·계산은 **DB에서 lookback만큼 읽어서** 사용 — API 재호출 없음

기존 `data/*.py` fetcher들의 "API에서 가져오기" 로직은 그대로 두고, 위에 신규 `data/store.py`
모듈 하나가 "DB확인 → 갭만 fetch → DB저장 → DB에서 읽기"를 감싼다. v3의 OpenBB 전환이 일어나도
`fetch_fn`의 내부 구현만 바뀌는 것이라 이 계층은 영향받지 않는다.

## 5. 갱신 주기

모든 소스가 일별 빈도라 하루 한 번 갱신이면 충분하다. 두 가지 방식:

- **A. 앱 기동 시 체크** — Streamlit 앱이 켜질 때 "오늘 데이터 있나? 없으면 갭만 fetch" 한 번
  실행. 별도 인프라 불필요, 첫 접속자가 약간의 지연을 감수.
- **B. 별도 스케줄러** — Windows 작업 스케줄러로 새벽에 `python ingest.py` 한 번 돌려 DB를
  미리 채워둠. 앱은 항상 DB만 읽으므로 응답이 빠름.

권장: 처음엔 A로 단순하게 시작하고, 실사용 트래픽이 늘면 B로 분리.

## 6. 이 구조가 직접 푸는 기존 한계

| 기존 한계 | 해결 |
|---|---|
| 기획안.md §11 "Block G는 현재 EPS로 과거를 되돌아본 것" | `manual_override_history`로 과거 EPS 변경 시점·값 추적 가능 |
| RRG가 현재 시점 점 하나만 표시 | `sector_metrics_daily` 누적되면 trailing tail 가능 |
| v3 walk-forward/이벤트스터디/Triage 검증 데이터 부재 | `series_daily`+`scores_daily`+`watchlist_events`가 정확히 그 입력 |
| H. 데이터 신선도 바가 세션 내 fetch 시점에만 의존 | `SELECT MAX(date) FROM series_daily WHERE series_id=?`로 재시작에도 살아남는 신선도 체크 |
| 캐시 만료마다 전체 lookback 재요청 | 증분 fetch로 API 호출량 대폭 감소 |

## 7. 도입 단계

1. `data/store.py` 신규 — SQLite 연결·5개 테이블 생성(`CREATE TABLE IF NOT EXISTS`)·
   `ensure_series_up_to_date()`/`upsert_*()`/`query_*()` 헬퍼.
2. 기존 fetcher 호출부(`app.py`의 `cached_*` 함수들)를 `store.py` 경유로 교체 — 한 시리즈씩
   점진적으로(예: 먼저 지수레벨 하나만 바꿔서 검증 후 나머지).
3. 첫 부트스트랩 실행 — 기존과 동일한 lookback(365일 등)으로 DB를 한 번 가득 채움.
4. Block G(히스토리)·H(신선도)를 DB 읽기 기준으로 전환 — 라이브 fetch 의존 제거.
5. `scores_daily`/`sector_metrics_daily`/`watchlist_events` 일별 기록 추가 — 매 세션 계산 후
   "오늘 날짜로 이미 기록 있나" 체크하고 없으면 insert.
6. (선택) RRG 트리맵/차트를 DB의 trailing 데이터로 확장.

각 단계는 기존 빌드 단계들과 동일하게 **synthetic 검증 → 실데이터 검증 → 서버기동 확인**을
거칠 것.

## 8. 리스크 / 주의사항

- **SQLite 동시쓰기 제한** — Streamlit 멀티세션에서 동시 쓰기가 겹치면 락 경합 가능. `PRAGMA
  journal_mode=WAL` 켜고, 가능하면 쓰기는 ingest 스크립트가 전담 + Streamlit 앱은 읽기 전용으로
  운영하는 게 가장 안전.
- **초기 백필 비용** — 처음 DB를 채울 때는 기존과 동일하게 무겁다(365일 전체). 그 다음부터만
  가벼워진다.
- **series_id 네이밍을 처음에 잘 정할 것** — 예: KOSPI 지수는 pykrx/yfinance 어느 쪽이 채우든
  동일한 논리 식별자(`KOSPI_INDEX`)를 써야 한다. 나중에 바꾸면 마이그레이션이 필요하다.
- **`dashboard.db` 파일은 `.gitignore`에 추가** — config.json/sectors.json과 달리 이건 매일
  자동 생성되는 데이터라 git에 커밋하지 않는다.
- **보존 정책은 당장 불필요** — 이 데이터량(시리즈 100개 × 일별)은 수년치도 SQLite 파일 기준
  수십 MB 수준이라 삭제 정책을 지금 정할 필요는 없다.
