# 유동성 천장 모니터링 대시보드 — 통합 기획안 (v2)

> v1(원 스펙)은 "여력(유동성 천장)" 레이어 하나만 다뤘다. 실제 프로토타입을 검토하면서 그 위·아래로
> **흐름(Flow)** 레이어와 **주도(Leadership: 섹터·종목)** 레이어가 빠져 있다는 게 드러났고, v2는 이
> 셋을 하나의 4단 모델(흐름▸여력▸주도▸정합성)로 통합한다. 이 문서는 Streamlit으로 실제 구현해
> Phase 1~7을 전부 동작 확인한 뒤 정리한 것이라, "이론상 이렇게 될 것"이 아니라 "실제로 이렇게
> 막히고 이렇게 풀었다"는 내용이 §11에 포함돼 있다. 다른 사람이 구현한다면 §11을 가장 먼저 읽을 것.

## 0. 한 줄 철학

천장 숫자를 맞히는 도구가 아니다. **지금 무슨 일이 일어나고 있고(흐름) → 그걸 받칠 연료가
얼마나 남았고(여력) → 연료가 지금 어디로 흐르고 있고(주도) → 가격이 그 연료보다 앞서 있는지
뒤처져 있는지(정합성)** — 이 네 가지를 한 화면에서 동시에 보는 것이 핵심이다. 천장 밴드·여력
점수·주도주 랭킹은 시나리오 프레임을 위한 정성·정량 산출물이며 예측이 아니다. 투자 판단과
책임은 사용자 본인에게 있다.

---

## 1. 핵심 모델 — 4개 레이어

```
지수 = 선행 EPS × 멀티플(PER)
유동성 천장 = 선행 EPS × [현재 유동성이 지지 가능한 최대 멀티플]
```

| 레이어 | 질문 | 핵심 산출 |
|---|---|---|
| **① 흐름 (Flow)** | 지금 지수가 어디서 어떻게 움직이는가 | 현재가·1일%·YoY%·선행/trailing PER·breadth·변동성 |
| **② 여력 (Liquidity)** | 그 움직임을 받칠 연료가 얼마나 차 있는가 | 시나리오별 천장 밴드(base/bull/hyper) + 6원천 합성점수(regime) |
| **③ 주도 (Leadership)** | 연료가 지금 어느 섹터·종목으로 흐르는가 | 섹터 트리맵(시총×YTD) + RRG 순환매(RS-Ratio×RS-Momentum) + key player/rising star |
| **④ 정합성 (Reconciliation)** | 가격이 연료보다 앞서 있는가 뒤처져 있는가 | 가격이 속한 밴드 vs regime이 허용하는 최고 밴드 비교 → 정합/과열/여유 |

②는 v1 스펙의 본문 그대로(§0/§4와 동일한 수식), ①·③은 v2 신규, ④는 v1에 있었지만 v2에서
③의 산출물(주도 섹터)까지 같이 요약하는 **크로스 내러티브 배지**로 승격된다.

## 2. 대시보드가 답하는 질문 → 출력 매핑

| 질문 | 출력 컴포넌트 |
|---|---|
| 지금 무슨 상황인가 (흐름) | KPI 스트립(1일%/YoY%/PER/breadth/vol) + 스파크라인 |
| 천장이 어디인가 (여력) | 시나리오별 천장 밴드 차트 |
| 연료가 얼마나 찼는가 (여력) | 6원천 패널 + REGIME 게이지 |
| 연료가 어디로 흐르는가 (주도) | 섹터 트리맵 + RRG 순환매 + key/star 카드 |
| 가격이 앞섰는가 뒤처졌는가 (정합성) | 정합성 배지 + 크로스 내러티브 배지 |
| 다음 위험 신호는 무엇인가 | 워치리스트 |
| 지금이 역사적으로 어디인가 | 히스토리 오버레이 |
| 데이터가 살아있는가 | 신선도 바 |

## 3. 화면 구성

상단 토글로 KOSPI / NASDAQ 전환. 6원천(S01~S06)은 **두 시장이 공유**하며 시장별 가중치만
다르다 — 섹터 트리맵/RRG는 시장별로 완전히 다른 데이터셋(§6).

```
┌─────────────────────────────────────────────────────────────┐
│ [KOSPI ▾]  흐름▸여력▸주도▸정합성 — 자동 생성 한 줄 배지       │ ← 크로스 내러티브
├─────────────────────────────────────────────────────────────┤
│ 현재 9,064(+0.8%,1D) │ YoY +199% │ 선행7.5x/trail11.2x │     │
│ Breadth 상승197·하락673 │ VKOSPI(근사)19.4   [스파크라인]   │ ← A. Flow KPI
│ 현재밴드 BASE │ 천장까지 +18%                                │
├──────────────────────────────┬──────────────────────────────┤
│ B. 천장 밴드 차트             │ C. 유동성 REGIME 게이지       │
│  hyper 14–16k ──────         │   ░░░░░░░▓▓ 38/100           │
│  bull  12–13k ──────         │   "BULL 연료 부분충전"        │
│  base  10.5–11k ●현재9,064   │   반영원천 6/6                │
├──────────────────────────────┴──────────────────────────────┤
│ E. 6개 원천 패널 — 현재값·여력바·방향·스파크라인 (S01~S06)    │
├──────────────────────────────┬──────────────────────────────┤
│ D. 정합성 배지: 🟡 여유       │ 유동성 지지 천장: 12,000       │
├──────────────────────────────┴──────────────────────────────┤
│ 주도(Layer3): 섹터 트리맵(크기=시총,색=YTD) │ RRG 순환매     │
│ [섹터 선택▾] → Key/Star 카드 → 종목 상세(테제·stats·risk)    │
├─────────────────────────────────────────────────────────────┤
│ F. 워치리스트 (지표·트리거·상태)                              │

│ G. 히스토리: 지수 추이 vs 천장 밴드 오버레이                  │
```

### 블록별 정의

- **크로스 내러티브 배지** — "흐름▸여력▸주도▸정합성" 한 줄. **고정 문구를 쓰지 않고 매번 라이브
  데이터에서 자동 생성**한다(흐름=breadth+YoY 텍스트, 여력=regime 라벨+천장까지%, 주도=RRG
  "leading" 1위 섹터+"improving" 1위 섹터, 정합성=배지 라벨). 화면 맨 위에 보여야 하지만 계산은
  모든 레이어 데이터가 다 모인 뒤에야 끝나므로, 구현 시 `st.empty()` 같은 플레이스홀더로 위치를
  먼저 잡고 나중에 채우는 패턴이 필요하다(§11).
- **A. Flow KPI** — 현재가(+1일%), YoY%, 선행PER(+trailing 보조), breadth, 변동성, 최근 추세
  스파크라인. 기존 v1의 "KPI 헤더"(현재밴드/천장까지여유)는 그대로 두고 그 **위에** Flow 메트릭을
  덧붙이는 구조로 합쳤다 — 두 블록을 따로 분리하면 "현재가"가 중복 표시된다.
- **B. 천장 밴드 차트** — v1과 동일(§0/§4-3).
- **C. 유동성 REGIME 게이지** — v1과 동일(§4-2). 단, 6원천 중 일부가 manual 미입력이면 **available
  한 원천만으로 가중치를 재정규화**하고 "반영원천 N/6"을 같이 표시해야 한다(전부 없으면 N/A,
  silent하게 틀린 점수를 보여주면 안 됨).
- **D. 정합성 배지** — v1과 동일(§4-3). 🟢정합/🔴과열/🟡여유 3상태 + above_hyper(천장 초과) 엣지케이스.
- **E. 6원천 패널** — v1과 동일. 각 카드: 현재값·스파크라인·방향 화살표·여력 점수바(0–100).
  manual 단일값 소스(신용잔고·MMF)는 시계열이 없으므로 스파크라인 없이 값+입력일만.
- **주도 (Leadership, 신규)** —
  - **섹터 트리맵**: Plotly Treemap. 크기 = 추적종목 시가총합, 색 = YTD%(RdYlGn, 0% 중심).
  - **RRG 순환매**: RS-Ratio(상대강도, x축) × RS-Momentum(모멘텀, y축), 100 중심 4분면
    (leading/weakening/improving/lagging). 계산법은 §5-2.
  - **종목 카드**: 섹터 선택 시 그 섹터의 key player·rising star 카드(티커·실시간가·YTD) +
    선택한 종목의 테제/stats/risk 상세. 테제는 **수동 큐레이션 콘텐츠**(§6).
- **F. 워치리스트** — v1과 동일(§7), 단 자동판정 불가능한 항목(MMF 주간추이, breadth 신고가
  다이버전스)은 "수동확인 필요" 상태로 명시(§11).
- **G. 히스토리** — 과거 지수 레벨 vs **현재 입력된** forward EPS 기준 천장 밴드 오버레이.
  과거 EPS 추정치 변화는 반영하지 않는다(과거 EPS 시계열이 없음 — 한계로 명시).
- **H. 데이터 신선도 바** — 지표별 출처·갱신주기·최근갱신일·staleness 색상(🟢최신/🟡갱신필요/⚪미입력).
  manual 콘텐츠(섹터 테제 포함)도 빠짐없이 한 줄씩 들어가야 한다.

가중치(KOSPI: 외국인·중앙은행 핵심 / NASDAQ: 중앙은행·대기자금·글로벌신용 핵심)도 v1과 동일.

### 5-2. RRG (순환매) 계산 — v2 신규

공식 JdK RS-Ratio/Momentum의 단순화 버전. 섹터 시계열과 벤치마크(KOSPI=`^KS11`,
NASDAQ=`^IXIC`)가 있으면:

```
relative        = 섹터시계열 / 벤치마크시계열
RS-Ratio(t)      = 100 × relative(t) / relative.rolling(ratio_window).mean()
RS-Momentum(t)   = 100 × RS-Ratio(t) / RS-Ratio(t - momentum_window)
```

둘 다 **100을 중심**으로 한다. 4분면: RS-Ratio≥100 & RS-Momentum≥100 → leading,
≥100 & <100 → weakening, <100 & ≥100 → improving, 나머지 → lagging.
`ratio_window`/`momentum_window`(기본 10거래일)는 config로 뺄 것 — 하드코딩 금지 원칙이
새 계산식에도 그대로 적용된다.

> 주의: RS-Momentum은 "방향"이 아니라 "가속도"를 본다. 선형으로 꾸준히 outperform하는 섹터는
> momentum이 100 근처로 평평해져 leading이 아니라 weakening으로 분류될 수 있다 — 버그가
> 아니라 RRG 지표의 정상적인 특성이다.

### 5-3. 섹터 가중치 / YTD

- **YTD%** = (최신가 / 연초가 − 1) × 100, 섹터 시계열에서 그대로 계산.
- **가중치**: 진짜 "시장 전체 대비 비중"은 분모(전체 시장 시총)가 신뢰도 낮아(특히 NASDAQ
  GICS ETF는 S&P 기준이라 NASDAQ 전체 시총과 분모가 안 맞음), **트리맵 sizing엔 추적 종목
  시총의 절대 합산값**을 그대로 쓰고 "추적 종목 기준"이라고 명시한다. 진짜 시장점유율이
  필요하면 별도 분모 소스를 구해야 한다.

## 6. 섹터·리더십 콘텐츠 아키텍처 (v2 신규, 가장 중요한 설계 결정)

이 레이어의 본질적 문제: **"반도체가 견인하고 있다"는 사실은 라이브로 계산되지만, "왜
한미반도체가 rising star인지"의 투자 테제는 계산되는 값이 아니라 사람이 쓰는 글**이다. 두
종류를 명확히 분리해야 한다.

| 구분 | 예시 | 소스 |
|---|---|---|
| **계산되는 값** | 가격, YTD%, RS-Ratio/Momentum, 시가총액 | 라이브 (yfinance) |
| **수동 큐레이션 값** | 섹터 테마 그룹 정의, 종목→테마 매핑, key/star 태깅, 테제/stats/risk 텍스트 | `sectors.json` (수동, forward EPS와 같은 대우) |

### 섹터 분류 기준

- **NASDAQ**: GICS 11개 표준 섹터 = SPDR 섹터 ETF(XLK/XLF/...)와 1:1 매칭 → **완전 자동**,
  매핑 유지보수 불필요.
- **KOSPI**: "반도체"/"전력기기"/"화학·2차전지" 같은 분류는 KRX 공식 업종(전기전자, 화학 등
  33개)과 다른 **투자테마 그룹**이다. 공식 업종엔 없으므로 종목→테마 매핑을 수동으로 관리해야
  한다(`sectors.json`의 `tickers` 배열). 대안으로 KRX 공식 업종을 그대로 쓰면 매핑 유지보수는
  없어지지만 "반도체"가 "전기전자"에 묻혀 단독으로 안 보인다 — **트레이드오프이니 구현 전에
  팀과 합의할 것**.

### `sectors.json` 스키마

```jsonc
{
  "KOSPI": {
    "benchmark": "^KS11",
    "sectors": {
      "semi": {
        "name": "반도체",
        "tickers": ["005930.KS", "000660.KS", "042700.KQ", "058470.KQ"],  // yfinance용, KR은 .KS/.KQ 접미사 필수
        "names": { "005930.KS": "삼성전자", ... },        // 모든 추적종목의 표시명 (key/star 여부 무관)
        "key": ["005930.KS", "000660.KS"],                // 대형/벨웨더
        "star": ["042700.KQ", "058470.KQ"],                // 신흥/소형 모멘텀
        "leaders": {                                        // key+star만 — 수동 큐레이션 콘텐츠
          "005930.KS": {
            "role": "메모리·파운드리 벨웨더",
            "thesis": "투자 테제 한두 문장",
            "stats": [["라벨", "값"], ...],
            "risk": "핵심 리스크 한 줄",
            "as_of": "2026-06-20"                           // staleness 추적용
          }
        }
      },
      "sec": { "name": "증권", "tickers": [...], "names": {...}, "key": [], "star": [], "leaders": {} }
      // key/star 없는 섹터는 names만으로 "주요종목: A·B·C" 표시
    }
  },
  "NASDAQ": {
    "benchmark": "^IXIC",
    "sectors": {
      "Tech": { "name": "Technology", "etf": "XLK", "tickers": [...], "names": {...}, "key": [...], "star": [...], "leaders": {...} }
    }
  }
}
```

- `etf` 필드가 있으면(NASDAQ) 섹터 시계열 = ETF 시계열 그대로. 없으면(KOSPI) `tickers`를
  각각 정규화(연초=100)한 뒤 **동일가중 평균**으로 섹터 대리 시계열을 만든다(시총가중 아님 —
  일별 시총 히스토리가 없어서 단순화한 것, §11에 명시).
- `leaders`는 key+star 종목만 채운다. 단순 섹터(증권/자동차 등)는 `names`만 있고 `leaders`는
  비워둔다 — 테제 없는 종목을 억지로 만들어내지 않는다.
- `as_of`로 콘텐츠 staleness를 신선도 바(H)에 노출한다. **라이브 갱신이 아니므로 분기 단위로
  사람이 다시 읽고 갱신해야 한다.**

## 7. 워치리스트

v1 §5와 동일한 7개 트리거. 구현 시 자동판정 가능 여부를 명확히 구분해야 한다:

| 트리거 | 자동판정 가능? |
|---|---|
| Fed 점도표/WALCL | 부분(WALCL 추세는 자동, 점도표 해석은 manual bool) |
| 외국인 3일 연속 순매도 | 자동 (일별 시계열 있음) |
| 신용융자 반대매매 | manual bool 플래그로만 (이벤트성, API 없음) |
| MMF 주간 감소 전환 | **수동확인** — manual_overrides가 단일 시점값이라 주간 비교 불가 |
| USD/KRW 1,500 하회 | 자동 |
| HY OAS/VIX 동반 확대 | 자동 (두 시계열 다 있음) |
| breadth 신고가 다이버전스 | **수동확인** — 52주 신고가 추적 미구현 |

MMF/breadth 다이버전스를 자동화하려면: MMF는 manual_overrides를 단일값 대신 **타임스탬프
배열**로 바꿔야 하고, breadth 다이버전스는 52주 신고가 종목수 집계 로직을 추가해야 한다 —
v2 범위 밖, 차기 과제로 남겨둠.

## 10. 빌드 단계 (실제로 동작 확인된 순서)

1. **골격 + 무료 API** — config 로더 + FRED 5종 + 지수레벨(pykrx→yfinance 폴백) + 천장밴드차트(B) + KPI헤더(A)
2. **KOSPI 수급** — 외국인 순매매 + breadth + 6원천 패널(E) 점등(미점수화)
3. **스코어링** — S01~S06 서브점수 → 합성 → regime게이지(C) + 정합성배지(D) + 워치리스트(F)
4. **히스토리·신선도** — 멀티플 시계열 오버레이(G) + staleness바(H)
5. **Flow 레이어** — 1일%/YoY%/trailing PER/breadth/vol + 스파크라인 (기존 KPI헤더 확장)
6. **섹터 트리맵 + RRG** — `sectors.json` 설계 + RS-Ratio/Momentum 계산 + Plotly 트리맵/스캐터
7. **Key/Star 카드 + 크로스 내러티브 배지** — 섹터 선택 UI + 테제 상세 + 4레이어 요약 배지

각 Phase 끝에 **synthetic 데이터로 함수 단위 검증 → 실제 네트워크 호출로 엔드투엔드 검증 →
Streamlit 서버 기동 확인(HTTP 200)** 3단계를 거쳤다. 인증이 막힌 경로(pykrx)는 "무크래시
폴백"까지가 검증 범위였고, 인증 자체는 검증 못 함 — KRX_ID/KRX_PW 확보 후 재확인 필요.

## 11. 구현 시 반드시 알아야 할 것 (하드 러슨)

### 11-1. pykrx는 더 이상 무인증으로 안 된다

최신 pykrx(1.2.8 기준)는 `data.krx.co.kr` 호출에 `KRX_ID`/`KRX_PW` 환경변수 기반 로그인을
시도한다. 미설정 시 모든 `get_index_ohlcv`/`get_market_trading_value_by_date`/
`get_market_cap_by_ticker`/`get_market_ohlcv_by_ticker`/`get_index_fundamental` 호출이
**"Expecting value: line 1 column 1" JSON 디코드 에러로 조용히 실패**한다(예외는 던지지만
크래시는 아님 — try/except로 잡힘). data.krx.co.kr 무료 가입 후 `.env`에 채워야 외국인
수급·시총·breadth·trailing PER이 라이브로 산다. **인증 없이도 되는 건 지수 레벨/개별종목가
(yfinance 폴백)뿐**이라고 가정하고 설계할 것.

### 11-2. yfinance가 의외로 KR 개별종목까지 된다

`005930.KS`(코스피)/`042700.KQ`(코스닥) 같은 접미사로 yfinance가 인증 없이 정상 응답한다.
이 덕분에 **섹터 트리맵·RRG·key/star 카드 전체가 KRX 인증과 무관하게** 구현·검증 가능했다.
처음엔 KOSPI 섹터지수를 KRX 서브인덱스 코드로 찾으려 했는데, 그게 인증 필요라는 걸 알고
나서 "개별종목을 yfinance로 모아 합성 섹터지수 만들기"로 방향을 틀었다 — 더 간단하고 더
검증 가능했다.

### 11-3. §4-1 수식이 요구하는 입력 중 §3 데이터 테이블에 없는 게 있다

Fed 점도표 인하여부, 시장 인상확률(CME FedWatch류), 외국인 보유비중 z-score — 셋 다 무료
API가 없어 결국 manual_overrides로 갔다. 특히 외국인 보유비중은 §3가 약속한 "외국인
순매매"(흐름 데이터, pykrx로 됨)와는 **다른 지표**(보유 레벨)라는 걸 헷갈리기 쉽다. 처음
스펙 읽을 때 "외국인 데이터 있으니 자동되겠지"라고 생각하면 틀린다.

### 11-4. VKOSPI 같은 한국 자체 변동성 지수는 무료 소스가 없다

VIX(미국)는 FRED `VIXCLS`로 바로 되지만 한국엔 그런 무료 공식 소스가 없다. **20일 실현변동성
(연율화)으로 근사**했고 UI에 "VKOSPI(근사)"로 명시했다 — 프로토타입 작성자도 이미 "근사"라고
적어놨던 걸 보면 의도된 타협이다.

### 11-5. KOSPI "단순 섹터"의 종목 매핑은 일반 지식으로 채웠다 — 검증 안 됨

증권/자동차/화학/바이오/은행/조선/통신/유통 섹터의 대표 3종목 티커는 LLM의 일반 지식으로
매핑했다(예: 한국금융지주=071050). KRX 인증 붙으면 **실거래 데이터로 한 번 대조 확인할 것** —
틀린 티커가 하나라도 있으면 엉뚱한 종목의 가격이 표시된다.

### 11-6. RS-Momentum은 "방향"이 아니라 "가속도"다 (§5-2 참고)

처음 단위테스트에서 "꾸준히 outperform하는 섹터"가 leading이 아니라 weakening으로
나와서 버그로 의심했는데, 가속/감속 곡선으로 다시 테스트해보니 4분면이 전부 정상 작동했다.
RRG를 처음 구현하는 사람은 같은 의심을 할 가능성이 높다 — 미리 알아두면 시간 절약된다.

### 11-7. 크로스 내러티브 배지는 화면 위, 계산은 맨 마지막

화면 최상단에 노출해야 하는데 계산 재료(흐름/여력/주도 전부)는 스크립트 맨 마지막에야
다 모인다. Streamlit이라면 `st.empty()`로 자리를 먼저 잡고 나중에 `.container()`로 채우는
패턴이 필요하다. React/다른 프레임워크로 갈 경우 이 제약이 없으니 레이아웃 순서를 그대로
컴포넌트 트리 순서로 짜면 된다 — Streamlit 특유의 제약이라는 점만 알아두면 됨.

### 11-8. 콜드 캐시 첫 로드가 느리다

섹터당 3~5개 티커 × 10~19개 섹터 = 시장당 30~45개 yfinance 개별 호출. 캐시 없이는 시장당
20~25초. ttl=3600 캐시로 시간당 1회만 이 비용을 낸다 — 사용자 수가 많아지거나 새로고침
주기가 짧아지면 재검토 필요(예: 백그라운드 스케줄러로 미리 채워두기).
